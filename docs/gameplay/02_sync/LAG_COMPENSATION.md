> **用途**：定义 Atlas PvP 中"如何在网络延迟下保持战斗公平"的设计哲学与具体规则。本文从**玩家公平感受**的角度展开——为什么需要补偿、补偿带来的副作用、玩家如何理解、何时拒绝补偿。

## 1. 为什么需要延迟补偿

### 1.1 一个简单的场景

PvP 1v1 对决中，玩家 A（ping 80ms）攻击玩家 B：

```
真实时间轴（服务端权威）:

T=0 ms:   B 在位置 P
T=40 ms:  B 移动到位置 Q（500ms 内 1 米的速度）
T=80 ms:  A 攻击事件到达服务端
```

A 的客户端**屏幕看到的**：
- A 看到的 B 位置 = T=0 时的 P（因为远端实体延迟约 100ms）
- A 在屏幕上看见 B 在 P，瞄准 P，按下攻击键
- 输入到达服务端时，B 已在 Q

如果服务端按 T=80 时的 B 位置（Q）判定：
- A 屏幕看到打到 B（在 P 的地方）
- 服务端说 miss（B 已不在 P）
- **玩家感觉"明明打到了为什么 miss"**

这是网络游戏 PvP 公平性的最大难题。

### 1.2 解决思路

服务端**回溯 B 在 A 感知时刻的位置**：
- A 感知 B 在 P 的时刻 = T=0
- 服务端查询 B 在 T=0 的位置 = P
- 用 P（而不是 Q）做命中判定
- A 屏幕打到了 → 服务端确认命中

这就是**延迟补偿**——让射手的命中判定基于"他看到的"而不是"现在的"。

### 1.3 业界先例

延迟补偿不是 Atlas 独创：
- **Counter-Strike**（2000）：首先大规模使用，~200 ms 窗口
- **Source Engine** 系列：通用延迟补偿框架
- **Overwatch**：~250 ms 补偿
- **BDO**：早期 ~300 ms，后期收紧到 ~200 ms
- **Dragon Nest**：PvP arena 严格补偿

Atlas 沿用业界共识但针对动作 ARPG 调优。

## 2. 补偿的玩家体验

### 2.1 射手（Attacker）视角

**正常体验**：
- 我看到敌人在那 → 攻击 → 打到了 → 伤害生效
- 没有"明明命中却 miss"的怒气
- 反应快的玩家有奖励（看准就打中）

**极端情况**：
- 极高 ping（> 200ms）：超出补偿窗口 → 可能 miss
- 这是合理的——你网络不行，服务端不能为你过度倾斜

### 2.2 受击者（Target）视角

**接受的延迟体验**：
- 在我屏幕上"我已经躲开了"，但还是被打到
- "他怎么打中的我？我都跑出 1 米了！"

这种体验**心理上不舒服**，但是**公平的代价**：
- 攻击者基于他看到的位置发起攻击
- 受击者基于他自己看到的位置躲避
- 两人各自的延迟使他们看到不同时刻
- 最终判定必须**取一方为权威**——选择射手（attacker）方

如果选择受击者方（"你以为打到了，但其实没")：
- 攻击者怒火更大（"我明明瞄准了！"）
- 玩家流失更多
- 商业角度更糟

业界共识：**Favor the shooter**（射手优先）。Atlas 沿用。

### 2.3 设计的心理学

让玩家**心理上接受**延迟补偿：
- 教学 / 入门里说明"延迟补偿是 PvP 公平的一部分"
- UI 显示双方 ping，让玩家理解差异
- 极端不公平案例提供录像查看
- 不要让玩家怀疑"游戏有 bug"——他们应该理解"这是规则"

### 2.4 高 ping 玩家的差异体验

ping 250 ms 的玩家：
- 上限 200ms 补偿可能不够
- 偶尔 miss 自己以为打到的目标
- PvP 体验明显差于低 ping 玩家
- **建议优化网络环境** 而不是抱怨补偿不够

PvE 不补偿，影响很小（NPC 不"狡猾"地躲）。

## 3. 补偿的规则细节

### 3.1 补偿窗口：200 ms

为什么是 200 ms：
- < 200 ms：覆盖大多数玩家（约 95% RTT < 200ms）
- > 200 ms 玩家本来 PvP 体验就差，补偿过多 = 让对方"被打到看不见的子弹"
- BDO / Overwatch 等都在这个区间

200 ms 是**经验值**——可调，但建议固定避免玩家适应混乱。

### 3.2 补偿计算（参见 `HIT_VALIDATION.md §5.4`）

```
A 感知到 B 的时刻 = 服务端当前时刻
                  − A 上行延迟 (RTT/2)
                  − A 远端实体显示延迟 (interp_delay ≈ 100ms)
                  − A 输入到 hitbox 触发延迟 (1 tick ≈ 33ms)
                  
                  → 大约 服务端当前 - (RTT/2 + 130ms)

补偿后位置 = B 的位置历史中"那个时刻"的位置
```

举例：A ping 60ms，A 感知 B 大约在服务端时刻前 (30 + 100 + 33) = 163ms。
服务端用 B 的 163ms 前位置做命中判定。

### 3.3 favor-the-shooter 边界容忍

参见 `HIT_VALIDATION.md §5.5`：在形状边界 0.2 m 内"接近命中"也算命中。

理由：
- 远端实体精度有限（位置 ±10 cm）
- 客户端预测有微小偏差
- 边界 5% 的容忍让"几乎打到"算数

效果：
- 攻击者：边缘擦剑也算命中（更爽）
- 受击者：极偶尔被擦边——可接受（不会频繁）

### 3.4 PvP 与 PvE 的差异

| 场景 | 补偿规则 |
|---|---|
| **玩家 → 玩家**（PvP arena） | 完整补偿 + favor-the-shooter |
| **玩家 → NPC** | 不补偿（NPC 没 ping，无意义） |
| **NPC → 玩家** | 不补偿（玩家本来就看到 NPC 当前位置） |
| **NPC → NPC** | 不补偿 |

简化模型：仅 PvP 才补偿。

### 3.5 iframe 检查时序

被补偿者在补偿瞬间已开 iframe（翻滚）→ **算闪避成功，不算命中**。

```
补偿时刻 = T=163ms 前
B 在 T=163ms 前的位置：原地未动
B 在 T=163ms 前的状态：iframe 已开（翻滚开始）

→ 命中判定：iframe 中，命中失败
```

这保证"我看到我躲了"的体验是真实的——iframe 是无敌的，无论攻击者基于何时位置判定。

详见 `HIT_VALIDATION.md §6.3`。

### 3.6 补偿时刻的状态可信性

服务端不仅记录历史位置，还记录历史状态（参见 `MOVEMENT_SYNC.md §7.2` PositionSample）：
- 位置
- 朝向
- iframe / Block / SuperArmor 状态
- HP

补偿时所有这些状态都用 "那时刻" 的值。

## 4. 玩家可能感到不公平的场景

### 4.1 "我躲了为什么还被打到"

**情况**：你看到敌人攻击，你按下翻滚 → 服务端判定击中。

**可能原因**：
- 你的翻滚输入到达服务端晚了（你 ping 高）
- 服务端基于攻击者感知时刻判定，那时你还没翻滚
- 攻击者的"补偿时刻"在你翻滚之前

**这是公平规则**：攻击者按他看到的时刻判定，而那个时刻你确实站着。

**减轻方法**：
- 你提前一点按翻滚（动作游戏的"读招" 技巧）
- 优化你的网络
- 或接受"我反应慢了" 这个事实

### 4.2 "我打到了为什么 miss"

**情况**：你瞄准敌人射击，屏幕看到命中 → 服务端判定 miss。

**可能原因**：
- 你的 ping 超过 200ms 补偿上限
- 敌人在你感知和服务端实际之间用了瞬移技
- 边界判定刚好不在 5% 容忍内

**应对**：
- 如果你 ping 高，优化网络
- 如果敌人用了闪现/瞬移，那是合理的
- 如果是边缘判定，考虑下次更精确瞄准

**不会出现**："我大幅度命中了却 miss"——这种情况不可能发生（除非 bug，需要报告）。

### 4.3 "我们互相打到了"

**情况**：A 和 B 几乎同时攻击对方，双方都受伤甚至双方死亡。

**可能性**：
- 同 tick 双方命中事件——按 EventId 顺序处理
- 因 Phase 6 死亡延迟（参见 `COMBAT_EVENT_ORDERING.md §3.3`），双方都能完成攻击
- 双方各自补偿后都"打到对方那时位置"

**这是设计意图**——动作游戏鼓励"同归于尽" 的紧张感。

### 4.4 "他闪现到我面前一击秒杀"

**情况**：敌人闪现技能后立即攻击，你没反应过来。

**这不是 lag comp 的问题**——闪现是合法位移技：
- 服务端权威下发位移
- 双方客户端按事件流同步
- 攻击在闪现到位后正常判定

PvE / PvP 都可能发生。这是**技能设计层面**问题，不是 lag comp。

### 4.5 "ping 时高时低，体验时好时坏"

**情况**：你的网络抖动，体验不稳定。

**自动调整**：
- jitter buffer 自适应（参见 `MOVEMENT_SYNC.md §6.6`）
- 抖动大 → buffer 扩大 → 视觉略滞后但稳定
- 抖动小 → buffer 收紧 → 视觉灵敏

**自适应的代价**：你的远端实体显示延迟变化，可能感觉"插值速度变了"。

**应对**：稳定网络（有线连接、关闭后台下载等）。

## 5. 客户端的视觉反馈

### 5.1 命中反馈

参见 `COMBAT_FEEL.md`：

攻击者命中确认（服务端 ack 后）：
- 准星 / 角色血条上显示"命中"标识
- Hit pause + camera shake
- 伤害数字弹出

**关键**：客户端**不预测命中视觉**——等服务端确认。
- 预测错（lag comp 后判 miss）→ 撤回视觉是糟糕体验
- 不预测：玩家略晚 50ms 看到反馈，可接受

### 5.2 受击反馈

被命中确认（服务端推送 Damage 事件）：
- 播放对应的 Hit Reaction 动画
- 屏幕红边脉冲
- HP 条减少
- 受击音效

**对受击者**："我屏幕上已经躲开了，怎么受伤了？" 的视觉感受会有：
- 短暂的"原地受击"动画（因为客户端按服务端事件触发）
- 玩家可能感到困惑

**改善**：
- 如果常发生，UI 可显示"被锁定" 提示，让玩家理解
- 整体 ping/jitter 监控良好时此情况罕见

### 5.3 ping 显示

UI 永远显示自己的 ping：
- 绿色：< 80 ms（极佳）
- 黄色：80–150 ms（正常）
- 橙色：150–200 ms（边缘）
- 红色：> 200 ms（差）

ping 持续红色 5+ 秒：弹出 "Network warning" 提示。

### 5.4 PvP arena 双方 ping 显示

进入 arena 时显示双方 ping：
- "Player A (ping 50ms) vs Player B (ping 80ms)"
- 让玩家理解双方延迟差异
- 透明度比"隐藏 ping" 更好

### 5.5 录像与申诉

PvP 战斗结束后**自动录像保存** 7 天：
- 玩家可申诉"明明打到了为什么 miss"
- 录像可慢动作 / 单帧观看
- 客服根据录像 + 服务端日志判定
- 大多数申诉是玩家误解 lag comp 规则——回放说明清楚后多数玩家接受

## 6. 服务端实现要点（设计层面）

### 6.1 位置历史缓冲

每玩家维护 1 秒位置历史（参见 `MOVEMENT_SYNC.md §7.2`）：
- 30 Hz × 1s = 30 个 sample
- 每个 sample ~32 字节
- 总开销 ~1 KB / 玩家
- 5000 玩家共 5 MB

可忽略的开销，但**必须每 tick 写入**——丢失则补偿失效。

### 6.2 历史精度的权衡

存太频繁（每 tick）：精度高，开销 ~1 KB/玩家，可接受。
存太稀疏（每 5 tick）：精度低，节省内存，但补偿误差大（5 tick 内位置变化达半米）。

我们选**每 tick**，保证补偿精度 < 10 cm。

### 6.3 补偿计算的开销

每命中判定额外的开销：
- 查 sample（O(log n) 二分查找）：< 1 μs
- 插值位置（线性插值）：< 1 μs
- 总：每命中判定 < 5 μs（含状态查询）

不影响 hot zone 性能。

### 6.4 跨 cell 时的补偿

实体跨 cell 后，旧 cell 的位置历史丢失。但**战斗中实体锁 cell**（参见 `OVERVIEW.md §5.3`），战斗中位置历史完整。

罕见极端情况（被击退跨 cell 的瞬间）：补偿失败，按当前位置判定。极少出现，可接受。

### 6.5 反作弊

补偿机制可能被滥用：
- 假高 ping（实际网络好但报高 ping） → 获得更长补偿
- 服务端检测：实际包到达时间 vs 玩家声称 ping
- 不一致超过阈值 → 视为作弊

参见 `08_security/ANTI_CHEAT_STRATEGY.md`（待写）。

## 7. 与其他游戏的对比

### 7.1 Counter-Strike 系列

- 补偿窗口 200 ms
- 完全 favor-the-shooter
- 同样有"我躲了为什么被打到" 的争议

CS 玩家社区已普遍接受这种规则——20 多年的教育。

### 7.2 Overwatch

- 补偿窗口 ~250 ms
- favor-the-shooter
- 但有特殊处理（Tracer 闪现技能 lag comp 特殊化）

Atlas 简化：所有技能统一规则，不为特殊技能特化。

### 7.3 BDO

- 早期 lag comp 较松（300 ms+），导致"被打到的一方" 体验差
- 后期收紧到 ~200 ms
- Atlas 一开始就 200 ms，避免后期改动玩家适应混乱

### 7.4 Dragon Nest

- PvP arena 严格补偿
- PvE 不补偿
- Atlas 沿用此模式

### 7.5 fighting games（街霸、铁拳等）

- Rollback netcode：双方都假设对方动作，预测错则回滚到正确状态
- 极致公平但实现复杂，且仅适合 1v1 小尺度
- Atlas 不用 rollback netcode（不适合 MMO 尺度）

我们用"延迟补偿 + favor-the-shooter"——业界中庸但成熟。

## 8. 性能与扩展

### 8.1 5000 并发的开销

每玩家位置历史 ~1 KB，5000 玩家共 5 MB。

每命中判定补偿开销 ~5 μs。Hot zone 200 命中/秒 → 1 ms/秒。可忽略。

### 8.2 PvP arena 内的优化

PvP arena ≤ 8 玩家，全部补偿但开销 < 100 μs / tick。极小。

### 8.3 大规模 PvE 的处理

大规模 PvE（练级点 30 玩家 vs 50 怪物）：
- 玩家 → 怪物：不补偿，跳过开销
- 怪物 → 玩家：不补偿
- 玩家 → 玩家：可能（如果热情 PK），少见

总开销可控。

## 9. 客服与玩家投诉处理

### 9.1 常见投诉

按频率排序：
1. "我打到了对方但 miss"
2. "我躲了对方但被打中"
3. "明显感觉延迟"
4. "对方好像开挂"

### 9.2 处理流程

1. **检查录像**：服务端权威记录
2. **查 ping 数据**：投诉玩家 / 对方的 ping
3. **判断**：
   - 是 lag comp 正常机制 → 解释规则给玩家
   - 是网络问题 → 建议玩家优化网络
   - 是 bug → 上报工程修复
   - 是作弊 → 上报反作弊系统
4. **回复**：友好且明确，避免"模糊"应付

### 9.3 教育玩家

游戏内：
- 教学关讲解延迟补偿
- 设置面板有"网络体验" 解释页
- PvP 进入时显示规则

社区：
- 官方 wiki 有专门页面
- 视频教程展示规则
- 让玩家理解"这是规则，不是 bug"

## 10. FAQ 与玩家关注问题

### Q1: 延迟补偿对所有玩家公平吗？

绝对公平不可能（网络物理限制）。但是**相对公平**：
- 补偿让大多数（< 200ms ping）玩家体验一致
- 高 ping 玩家略差是可以接受的（他们本来就该提升网络）
- 低 ping 玩家稍微占优是合理的（他们的网络投资值得）

### Q2: 200 ms 是不是太严格？

国际化游戏可能要放宽（跨大洲玩家 RTT 高）。Atlas 暂定中国国服 200 ms（玩家 99% 在境内 ping < 150ms）。
未来海外服可能调到 250 ms。

### Q3: 我能否关闭延迟补偿，让游戏完全"公平"（不补偿）？

不可以——这会让玩家普遍体验"我打到了却 miss"，更糟。

延迟补偿的存在让大多数玩家的"打到"算数，是综合最优。

### Q4: 我是否可以选择"我打到的算数"，"我被打到的不算数"？

不可以——双方规则必须一致。否则攻击者可以"不接受我打中"，受击者可以"不接受我被打中"，每次命中都成谜。

### Q5: 服务器位置（北京 / 上海 / 广州）影响延迟补偿吗？

不影响补偿规则——但影响你的 ping。
低 ping = 补偿少 = 体验近"无延迟"
高 ping = 补偿多 = 体验"略受补偿"

选择就近服务器是明智的。

### Q6: PvE 中我也有"miss" 经历，那是什么？

PvE 不用 lag comp，但仍可能 miss：
- 你的攻击距离不够（hitbox 没覆盖）
- 怪物移动出 hitbox 范围
- 怪物 iframe / Block

不要怀疑 lag comp——PvE 没有它。

### Q7: 补偿时刻位置历史是回溯到那个时刻，那敌人状态变化怎么办？

按 §3.6——服务端记录历史包含状态（iframe / block）。补偿时用那时刻的状态判断 iframe / block。

例：你攻击 B → 补偿到 B 位置 P，但发现 B 在那时刻已开 iframe → miss。

### Q8: 如果我和敌人同时翻滚，谁的 iframe 优先？

各自的 iframe 在各自的时间轴生效。同 tick 双方都在 iframe → 双方都不会被命中。

### Q9: 双方 ping 差异极大（A 50ms，B 250ms）怎么办？

A 的攻击受补偿（B 在 200ms 前的位置）。
B 的攻击：自己的 250ms ping > 200ms 上限 → 补偿仅 200ms。
B 略吃亏，但**网络是 B 的责任**——不能让 A 因为 B 网络差而被惩罚。

### Q10: 服务器宕机或迁移时延迟补偿如何？

宕机：玩家断线，重新登录。
副本迁移（从一 CellApp 到另一 CellApp）：玩家短暂网络延迟（~50ms），但补偿规则不变。

非战斗时刻迁移（绝大多数情况），玩家完全无感。

### 反模式清单

- ❌ 让客户端发"我命中了" 给服务端（作弊门户）
- ❌ 不限补偿窗口（500ms+ 让被打的人崩溃）
- ❌ 补偿位置但不补偿状态（iframe 时序错乱）
- ❌ 仅补偿玩家 → 玩家，PvE 也补偿（PvE 无意义）
- ❌ 受击反馈预测（错了撤回视觉）
- ❌ 隐藏 ping 信息（玩家无法理解差异）
- ❌ 不提供录像（玩家投诉无据可查）
- ❌ 让 favor-the-shooter 容忍过宽（>10cm，被击的人发疯）
