# 实体快照（Entity Snapshot）

> **用途**：定义 Atlas 中"玩家如何看到其他实体"——位置 / 朝向 / 状态的快照协议、玩家视觉感受的设计取舍、不同场景下的呈现策略。本文从**玩家观看的角度**展开，不是协议手册。
>
> **读者**：网络工程师（必读）、客户端工程（必读）、战斗策划（§4 必读，理解远端实体表现限制）、技术美术（§5 必读）。
>
> **状态**：草案 v0.1 — 待团队评审。
>
> **前置文档**：`OVERVIEW.md`、`MOVEMENT_SYNC.md §6`（远端实体插值）、`DENSITY_ADAPTIVE_NETWORKING.md`（订阅分层）
>
> **关联文档**：`COMBAT_EVENT_ORDERING.md`（事件 vs 快照）、`GHOST_ENTITY.md`（服务端跨 cell）

## 1. 玩家看到什么

### 1.1 实体快照是什么

当玩家在游戏中看到**其他玩家或 NPC** 时，他们看到的不是"实时位置"，而是：
- **过去 100–120 ms 的位置**（远端实体延迟渲染）
- **服务端权威推送的快照**
- **客户端在两个快照之间用插值填充**

这是所有联网游戏的物理事实——网络延迟不可消除。我们的任务是**让玩家感觉不到延迟**。

### 1.2 玩家应该感受到的

✅ **流畅运动**：远端角色不抖动、不卡顿、不"跳格子"
✅ **跟手**：远端角色起步、急停、转向都自然
✅ **可读**：能从远端角色姿势/方向判断意图
✅ **位置可信**：靠近时位置精确（命中判定能命中）

### 1.3 玩家不应该感受到的

❌ **抖动**：每个快照导致角色"颤抖"
❌ **橡皮筋**：角色突然回弹/被拉回
❌ **果冻**：转向/急停时姿势漂浮
❌ **瞬移**：突然出现在新位置无过渡

### 1.4 设计的本质

**用过去信息预测当下，让玩家感觉"现在"** 是网络游戏的核心魔术。Atlas 的实现优雅与否决定 PvP 公平感和 PvE 沉浸感。

## 2. 快照内容（按订阅层）

### 2.1 与 DENSITY_ADAPTIVE_NETWORKING 的关系

参见 `DENSITY_ADAPTIVE_NETWORKING.md §3`：5 层订阅，每层数据精度不同。

本文不重复字段表，仅讨论**每层下玩家的视觉感受**。

### 2.2 L0 战斗圈（< 5 m, 30 Hz）

玩家**正在战斗或非常贴近**这些实体。

数据精度最高：
- 位置精度 ~1 cm
- 朝向 16-bit（0.0055°）
- 速度向量
- 当前动画状态简化标识

玩家感受：
- **战斗对手的剑刃位置精确**——hitbox 判定可信
- **闪避时机精准**——能精确预判敌人攻击的命中点
- **PvP 切磋**——双方位置 / 朝向 / 命中范围都清晰

### 2.3 L1 战术感知（5–15 m, 20 Hz）

玩家**视线主要焦点之外，但仍参与战斗**。

精度略低：
- 位置精度 ~10 cm
- 同样有速度
- 动画状态

玩家感受：
- **可看到队友/敌人在干什么**
- **能预判他们的下一步**（速度信息让插值平滑）
- **位置略不精确不影响战斗**——15m 外不会命中判定

### 2.4 L2 视野中（15–40 m, 10 Hz）

视野内但不在主战场。

精度更低：
- 位置精度 ~30 cm
- 无速度（用上一段速度估算）
- 朝向

玩家感受：
- **看到"那边有人在战斗"**
- **能识别角色 + 大概动作**
- **30 cm 偏差视觉不察觉**（远端角色看起来一致）

### 2.5 L3 视野边缘（40–80 m, 5 Hz）

远景人物。

精度更粗：
- 位置精度 ~1 m
- 方向用 8 个象限（前/右/后/左 + 4 个 45° 中间）
- 仅"是否在做事" flag

玩家感受：
- **看到"那边远处有人"**
- **大概知道在朝哪走**
- **位置 1m 偏差远距离视觉不察觉**——远端同样看起来一致

### 2.6 L4 远景存在感（80–200 m, 1 Hz）

地图远端的实体。

精度最低：
- 位置精度 ~3 m
- 仅位置，无其他

玩家感受：
- **看到"远方有人 / 怪物"**——存在感
- **不需要精确定位**（玩家走近时切到 L3 / L2，数据精度自动提升）

### 2.7 200m 外不订阅

玩家**根本看不到**：
- 客户端不知道这些实体存在
- 也无需广播（带宽节省）
- 玩家走近时实体进入 L4 → 渐显

## 3. 平滑插值的细节体验

### 3.1 Cubic Hermite 插值（参见 `MOVEMENT_SYNC.md §6.3`）

**为什么不用线性 lerp**：
- 线性 lerp 在拐弯处呈现"折线"——玩家感觉远端角色"机械"
- Hermite 用速度信息生成弧线——更像真实运动

玩家感受对比（远端玩家正在跑步转弯）：
- 线性：角色突然向新方向跳跃
- Hermite：角色平滑曲线转弯，像真在跑

### 3.2 急停的处理

远端玩家急停（速度从 5 m/s 降到 0）：
- 一个快照 velocity = 5 m/s
- 下一快照 velocity = 0
- Hermite 插值：在 50 ms 内平滑减速
- 视觉：角色微微"过冲" 然后稳住——和真实物理一致

### 3.3 急转向

远端玩家突然 180° 转向：
- 一个快照 yaw = 0°
- 下一快照 yaw = 180°
- yaw 插值用最短路径（不会绕远）
- 视觉：角色快速但平滑转身

### 3.4 抖动来源（应避免）

抖动通常来自：
- **快照频率太低**（如 5 Hz）+ 线性插值
- **未启用 jitter buffer**（直接消费收到的数据）
- **前后快照的速度大幅波动**

我们的做法：自适应 jitter buffer + Hermite 插值 → 抖动不可见。

### 3.5 玩家自己看到自己（主控）

主控完全不走快照——本地预测（参见 `MOVEMENT_SYNC.md §5`）。
玩家看到的"自己"是即时的，0 延迟。

## 4. 边缘场景的玩家体验

### 4.1 远端玩家闪现 / 瞬移

某玩家用闪现技能瞬移 20 米：
- 服务端发 `MovementCommand(Teleport)` 事件
- 客户端收到事件**立即停止插值**
- **直接跳到目标位置**（带瞬移特效）
- 视觉：清晰的"闪一下" 效果

不要试图"插值过去"——会显得角色突然变速移动，反而怪异。

### 4.2 远端玩家被击退

被击退跨过几米：
- 服务端发 `MovementCommand(Knockback, curve)` 事件
- 客户端收到事件后停止普通插值
- 按曲线平滑移动到目标位置
- 视觉：自然的击退弧线

参见 `MOVEMENT_SYNC.md §8` MovementCommand。

### 4.3 远端玩家死亡

死亡时刻的视觉：
- HP 降到 0 通过 `BuffApplied(Dead)` 事件传达
- 客户端立即停止 idle / locomotion
- 切到 death 动画
- 死亡动画 + ragdoll 接管

视觉：自然的"被打死" 表演，无瞬移、无僵硬。

### 4.4 远端玩家断线

某玩家断线：
- 服务端检测到（30s 超时）
- 实体进入"幽灵"状态（站立不动）
- 5 秒后实体消失（从其他玩家视野移除）
- 视觉：玩家站住，然后渐隐——比"突然消失"自然

### 4.5 包丢失或晚到

正常 UDP 网络下 < 1% 丢包：
- 单次丢包不影响（下一快照覆盖）
- 多次丢包导致快照间隙过大 → 客户端外推
- 外推 < 100 ms 用线性，< 300 ms 用减速衰减
- 超过 300 ms 实体冻结，等数据恢复

玩家感受：**网络好时完全无感**，**网络差时偶尔小卡顿但不撕裂**。

### 4.6 跨 cell 实体进入视野

玩家跑近一个 cell 边界，对面 cell 的实体进入视野：
- 服务端订阅扩展 → 推送实体快照
- 客户端从未见过此实体 → 从首个快照开始插值
- 第一帧实体出现位置 = 服务端推送的位置（无之前数据）
- 视觉：实体"渐显"或"突然出现"

可优化：渐显（fade in 200ms）让"突然出现"看起来不那么生硬。

## 5. 客户端 LOD 与视觉调和

### 5.1 距离与渲染细节

参见 `DENSITY_ADAPTIVE_NETWORKING.md §7`：

| 层 | 渲染策略 | 玩家所见 |
|---|---|---|
| L0 / L1 | 完整 mesh + 完整 animator + 完整 VFX | 看到所有细节 |
| L2 | 完整 mesh + 简化 animator + 减少 VFX | 看到大致动作 |
| L3 | 低模 + 静态摆姿势 + 极少 VFX | 看到"那边有人在动" |
| L4 | Billboard / Imposter | 看到"那边有个小点" |

### 5.2 切换平滑

玩家走近一个远端实体，从 L3 升级到 L1：
- L3：低模 + 摆姿势
- 玩家走近 → 进入 40m → 切到 L2 → mesh 升级
- 继续走近 → 15m → 切到 L1 → animator 启用

**视觉过渡 200 ms fade**：
- 旧 mesh 淡出
- 新 mesh 淡入
- 动画从 idle 开始
- 玩家感受：自然过渡，无突兀

### 5.3 远端 VFX 的简化

VFX 在远端简化原则：
- L0 / L1：完整 VFX（粒子全数）
- L2：仅主特效（保留视觉信号 + 减少粒子 50%）
- L3：极少特效或转 sprite（节省粒子 90%）
- L4：无 VFX

玩家感受：远处战斗的 VFX 仍能识别（蓝色 = 法师攻击，红色 = 火焰），但密度降低。

### 5.4 远端 Audio 的处理

距离衰减（3D 空间音效自然处理）：
- L0 / L1：完整音量 + 完整层
- L2：减弱（按距离衰减）+ 主音层
- L3 / L4：仅最关键事件（boss 大招的 audio cue 可远距离听到）

玩家感受：靠近能听见战斗细节，远了只听到模糊声音 + 重要 cue。

## 6. 不同游戏场景的取舍

### 6.1 PvP arena（1v1 / 2v2 / 4v4）

**场景特点**：8 人以下，单 instance，封闭空间。

**快照策略**：
- 全部实体在 L0 / L1（最高精度）
- 30 Hz 全量数据
- 无需分层（实体数 ≤ 8）
- 严格 Lag Compensation（参见 `LAG_COMPENSATION.md`）

**玩家感受**：
- 几乎"零延迟"感（< 100 ms 远端实体）
- PK 公平
- 命中精准

### 6.2 PvE 副本（4 / 8 人）

**场景特点**：玩家少，但怪物可能多（10–30 个）。

**快照策略**：
- 玩家 L0 / L1（高精度）
- 怪物 L1 / L2（中精度）
- 30 Hz 全部
- 远怪物 L3（低精度）

**玩家感受**：
- 队友清晰
- 怪物动作可读
- Boss 机制提示明确

### 6.3 开放世界主城

**场景特点**：玩家密集（50–200 人），但战斗少。

**快照策略**：
- 周围 5–10 玩家 L1（看着说话 / 互动）
- 中距离玩家 L2 / L3
- 远玩家 L4（"广场上人很多"的氛围）
- 20 Hz 主频率

**玩家感受**：
- 主城热闹，玩家不会"突然冒出来"
- 走近能识别个体
- 远处只是"人群"——不重要的精度

### 6.4 开放世界练级点

**场景特点**：怪物多（30–60），玩家适中（5–15）。

**快照策略**：
- 玩家全 L0 / L1
- 战斗中怪物 L1 / L2
- 远端非战斗怪物 L2 / L3
- 30 Hz（hot zone 提升）

**玩家感受**：
- 战斗清晰
- 怪物动作可识别
- 不卡

### 6.5 跨场景过渡

玩家从主城走到野外：
- 主城实体逐渐离开订阅
- 野外实体逐渐进入订阅
- 视觉：自然过渡，无 loading

跨大区域（用传送门进副本）：
- 旧场景所有实体退出订阅
- 加载副本场景
- 副本实体进入订阅
- 视觉：明确的 loading + 进入战斗

## 7. 性能与玩家可感受影响

### 7.1 客户端帧率

实体快照处理是相对轻量的工作：
- 接收 + 反序列化：< 100 μs / 包
- 插值计算：< 5 μs / 实体 / 帧
- LOD 切换：< 1 ms（200ms 一次）

50 实体同屏 + 60 FPS 渲染：插值占用 < 5%。

### 7.2 网络延迟容忍

| RTT | 玩家感受 |
|---|---|
| < 50 ms | 优秀，几乎察觉不到延迟 |
| 50–100 ms | 良好，正常体验 |
| 100–150 ms | 可玩，少数操作可能感到延迟 |
| 150–200 ms | 边缘，PvP 困难 |
| > 200 ms | 不推荐 PvP，PvE 仍可玩 |
| > 300 ms | 体验明显下降，需提示玩家 |

### 7.3 抖动容忍

| 抖动（标准差） | 玩家感受 |
|---|---|
| < 10 ms | 极佳 |
| 10–30 ms | 优秀 |
| 30–50 ms | 良好 |
| 50–100 ms | 可玩，jitter buffer 自动扩大 |
| > 100 ms | 不流畅，玩家明显感知 |

我们的 jitter buffer 自适应（80–250 ms）覆盖大部分抖动情况。

### 7.4 丢包容忍

| 丢包率 | 玩家感受 |
|---|---|
| < 1% | 无感 |
| 1–3% | 偶尔小卡顿 |
| 3–5% | 略明显抖动 |
| > 5% | 频繁卡顿，建议提示玩家 |
| > 10% | 几乎无法玩 |

## 8. 玩家提示与诊断

### 8.1 网络状态提示

UI 角落显示：
- Ping（数字 + 颜色：绿/黄/橙/红）
- 包丢失率（仅异常时显示）
- 服务器名

### 8.2 异常状态视觉

如果 RTT > 200ms 持续 5 秒：
- HUD 显示 "Network Issues" 警告
- 玩家可选"切换服务器"或继续

如果远端实体外推超过 300 ms（数据严重落后）：
- 该实体头顶显示"信号弱" 图标（其他玩家可见 = 公平信号）

### 8.3 PvP 公平性提示

PvP 匹配前提示：
- "您当前 ping XX ms，匹配低延迟服务器"
- 极高 ping 玩家不进低延迟队列

匹配后双方看到对方 ping，理解延迟差异。

### 8.4 录像与回放

战斗结束后玩家可看战斗录像：
- 录像基于服务端事件流（精确）
- 可以慢放、暂停、单步
- 用于学习和争议申诉

## 9. 服务端实现要点（设计层面）

### 9.1 Witness（订阅管理器）

每玩家在服务端有一个 Witness 对象：
- 维护"我能看见哪些实体"列表
- 每实体有当前订阅层
- 每 tick 决定本 tick 推送哪些实体的数据

详见 `DENSITY_ADAPTIVE_NETWORKING.md §5`。

### 9.2 Reference Position 机制

为了节省带宽，位置用 delta 编码：
- 周期推送一个"参考点位置"
- 后续位置存"相对参考点的偏移"
- 偏移用更少 bit 表达

每 2 秒或参考点失效（实体远离）刷新参考。

### 9.3 跨 cell 实体的快照来源

实体在 cell #1（Real），玩家在 cell #2：
- cell #2 的 Witness 订阅了实体（通过 Ghost）
- Ghost 数据由 cell #1 推送过来（参见 `GHOST_ENTITY.md`）
- Witness 用 Ghost 数据生成快照给玩家

跨 cell 不影响玩家的视觉体验——玩家不知道实体在哪个 cell。

### 9.4 优先级与丢弃

如果服务端 CPU 过载，可丢弃 L4 实体推送（仅推 L0 / L1 / L2 / L3）。
玩家感受：远景实体短暂"冻结"，几秒后恢复。

L0 / L1 永远不丢——这影响战斗判定。

## 10. FAQ 与玩家关注问题

### Q1: 为什么我看到的远端玩家位置和"实际"不一样？

因为实际位置和你看到的位置确实不一样——存在 ~100 ms 延迟。这是网络游戏的物理事实。

我们的目标是让你**感觉不到**这个差异，但极端情况（高 ping）下你可能注意到。

### Q2: 我感觉远端玩家有时"瞬移" 一下？

可能原因：
- 网络包延迟到达（jitter buffer 不够吸收）→ 调整网络
- 远端使用闪现技能（设计意图）
- 远端被强制位移技击中（设计意图）
- 远端跨 cell 出现（极少）

如频繁发生，请反馈服务器/网络问题。

### Q3: PvP 中"我打到了对方但没生效"是什么情况？

通常是延迟补偿的边界情况——参见 `LAG_COMPENSATION.md`。

简单说：你的攻击在你屏幕上看到命中，但对方那一刻已经移动开了。Atlas 优先你（射手），200ms 内的延迟补偿让你的命中算数。但超出 200ms 的极端情况下不补偿。

如果频繁发生，可能是你 ping 过高（> 200ms）。

### Q4: 为什么远端玩家有时"果冻"般晃动？

可能原因：
- 网络抖动严重（> 100ms 标准差）
- 客户端帧率不稳定
- 远端玩家网络问题

如果你的网络稳定但仍看到，可能是远端玩家网络问题——非你的问题。

### Q5: 远端玩家有时跑得快有时跑得慢？

服务端速度是固定的，差异来自插值：
- 网络好：完美 hermite 插值，速度恒定
- 网络抖动：插值校准时可能加速/减速一段以追赶时间
- 跨 cell 切换：可能有短暂调整

如果差异明显（如停顿后冲刺），通常是网络问题导致包堆积后释放。

### Q6: 100m 外的玩家位置精度只有 3m，对游戏体验有影响吗？

绝大多数情况无影响：
- 100m 外你看不清细节（即使位置精确）
- 不会影响战斗（你打不到 100m 外的人）
- 远距离技能（如远程射击）超出视距设计

PvP arena 内不存在 100m 外（场地小于 64m）。

### Q7: 服务器延迟更新远端玩家时，会让我"打不到"吗？

不会——本地预测让你的攻击立即播放动画，hitbox 在服务端按延迟补偿规则判定。补偿窗口 200ms 覆盖大多数情况。

只有当对方在 200ms 内做出大幅位移（闪现 / 极速移动），可能"打空"——这是对方的优秀操作，不是 bug。

### Q8: 服务端会对所有玩家广播相同数据吗？

不会——每个玩家的 Witness 独立维护订阅。视野不同 → 推送不同数据。

但**同一实体在不同玩家看来位置一致**（仅时间略有差异，因为各自延迟不同）。

### Q9: 移动端会不会因为带宽问题特殊处理？

Atlas 不支持移动端（参见 OVERVIEW §2.3）。桌面有线宽带条件下带宽 ~13 KB/s 单玩家完全可承受。

### Q10: 200v200 攻城战在 Atlas 行得通吗？

不做（参见 OVERVIEW §2.3）。200v200 是 BDO 战场尺度，对网络/CPU 极其挑战。Atlas 聚焦 1v1/2v2/4v4 小规模 PvP，质量优先。

### 反模式清单

- ❌ 不分层全订阅（带宽爆炸）
- ❌ 客户端 fixedUpdate 用 Time.deltaTime 插值（应用 server tick）
- ❌ 用线性 lerp 做位置插值（视觉差）
- ❌ 每个客户端推同样数据（无视个性化订阅）
- ❌ 实体瞬移用插值掩盖（应当感知，配上特效）
- ❌ 客户端"自己估算"远端实体位置（应只用服务端推送）
- ❌ 不限位置历史（无限内存）
- ❌ 远端 VFX 不做 LOD（性能炸）
- ❌ 网络异常无视觉反馈（玩家不知道发生了什么）
- ❌ 跨 cell 实体不渐显（突然出现破坏沉浸）
