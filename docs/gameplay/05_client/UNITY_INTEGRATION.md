## 1. 客户端的角色定位

### 1.1 一句话总结

**Unity 客户端是"玩家的眼睛和手"，不是游戏的真相**。

- 玩家通过 Unity 看世界（渲染、动画、UI）
- 玩家通过 Unity 操作角色（输入采集）
- 但游戏的"真实状态"——位置、HP、buff、命中——**永远在服务端**

### 1.2 这个定位决定了什么

**支持的玩家体验**：
- 即时输入响应（本地预测）
- 流畅视觉（动画/特效/相机）
- 优秀手感（hit pause / camera shake）
- 个性化设置（画质/UI/无障碍）

**坚定不做的**：
- 客户端权威伤害（防作弊）
- 客户端权威位置（防穿墙/瞬移挂）
- 客户端权威死亡（防"我看你死了你还在打我"）

### 1.3 与服务端的契约

```
┌─────────────────────────┬─────────────────────────────┐
│  Unity 客户端做         │  服务端做                    │
├─────────────────────────┼─────────────────────────────┤
│ 渲染（mesh / 材质 / 光照） │ 仿真（位置 / 战斗 / buff）   │
│ 播放动画（Animator）     │ 决定动画状态切换时机          │
│ 视觉/听觉反馈            │ 触发反馈事件                 │
│ 玩家输入采集             │ 验证输入合法性               │
│ 本地预测（手感）          │ 权威结算（公平性）            │
│ UI / HUD                │ 推送 UI 数据                 │
│ 音频混合                 │ 触发音效事件                 │
│ 相机控制                 │ 不参与                       │
│ 设置 / 无障碍            │ 不参与                       │
└─────────────────────────┴─────────────────────────────┘
```

## 2. Unity 提供什么，我们覆盖什么

### 2.1 我们用的 Unity 特性

| Unity 特性 | 用途 | 玩家可见的影响 |
|---|---|---|
| **渲染管线（URP/HDRP）** | 全部场景渲染 | 画质表现 |
| **Animator + Mecanim** | 角色动画播放 | 动作流畅度 |
| **Cinemachine** | 相机控制 | 视角自然、镜头切换 |
| **VFX Graph / Particle System** | 战斗特效 | 视觉冲击力 |
| **Input System (新)** | 输入采集 | 响应灵敏度、按键映射 |
| **AssetBundle** | 资源动态加载 | 进入副本的速度、热更体验 |
| **Audio System** | 音效播放 | 沉浸感 |
| **TextMesh Pro** | UI 文本 | UI 清晰度 |

### 2.2 我们覆盖的 Unity 特性

| Unity 默认行为 | Atlas 覆盖原因 | 替代方案 |
|---|---|---|
| **Rigidbody / CharacterController** | 非确定性，PhysX 跨端不一致 | 自研胶囊 + 高度图碰撞（C++ native） |
| **Physics.Raycast** | 同上 | 自研形状重叠（端共享） |
| **Animator Root Motion** | 运行时计算非确定 | 构建期提取曲线 |
| **NetworkManager** | 与 Atlas 协议不兼容 | 自研协议层 |
| **Time.timeScale 全局** | 影响所有系统不可控 | 仅 hit pause / slomo 局部使用 |
| **Random** | 端同破坏 | DeterministicRng |
| **Burst Compiler** | 默认 -ffast-math 破坏端同 | 仅用于非权威批处理（远端插值） |

### 2.3 玩家如何感知这些选择

正面：
- **战斗判定公平**——服务端权威的物理碰撞，没有"穿透剑刃"的物理 bug
- **跨平台/跨配置一致**——低端机 vs 高端机的玩家在战斗结果上没差异
- **网络韧性**——预测和解机制保证手感

负面：
- **加载略慢**：自研碰撞数据需独立加载（~50ms 进入战斗时多花费）
- **某些 Unity 教程不适用**：无法直接套用 Unity 标准物理流程

权衡**净正面**——MMOARPG 公平性远比开发便利重要。

## 3. 玩家体验关键路径

### 3.1 输入到反馈的闭环

```
玩家按下"攻击"键
   ↓ < 16 ms (Input System 采集)
客户端发送输入帧 + 本地播放挥剑动画 + 视觉占位 hitspark
   ↓ < 16 ms (帧渲染)
玩家看到角色挥剑（本地预测 = 0 延迟体验）
   ↓ ~50–100 ms (网络往返)
服务端确认命中 + 发送伤害事件
   ↓ < 16 ms (反馈触发)
客户端显示伤害数字 + hit pause + camera shake
```

**玩家感知的延迟**：
- 视觉响应：< 16 ms（极佳）
- 数字 / 反馈：100–150 ms（玩家几乎不察觉，因为有动画 / 音效占位）

### 3.2 远端实体的呈现

```
服务端 tick T：远端玩家 X 在位置 P
   ↓
位置快照通过网络发送
   ↓ ~50 ms 到达
客户端接收，存入 EventBuffer
   ↓ ~100 ms 后（interp_delay）
客户端在 X 的渲染时间到达 → 渲染于位置 P
```

玩家感知：远端玩家的动作总是"过去 100 ms 的动作"。这是**所有动作 MMO 的物理事实**，不可消除，只能藏好。

我们如何藏：
- 远端动画提前播放（attack 起手让玩家有反应时间）
- 远端命中事件按服务端 tick 排序（不会顺序混乱）
- 网络抖动时插值缓冲自动扩大（更稳）

### 3.3 性能预算（玩家可感）

| 项目 | 预算 | 玩家感受 |
|---|---|---|
| 帧率 | ≥ 60 FPS | 流畅 |
| 帧时间 stable | < 16 ms p99 | 不卡顿 |
| 输入到画面 | < 33 ms | 跟手 |
| 进入副本 loading | < 5 s | 不烦躁 |
| 资源热更下载 | < 30 s（增量） | 可接受 |
| 场景内存 | < 4 GB | 8 GB 机型可玩 |

**玩家会立即注意到**：帧率掉到 30、加载超过 10 秒、热更超过 5 分钟。这些是硬限制，运营时持续监控。

## 4. 主线程纪律

### 4.1 为什么重要

Unity 几乎所有 API 必须在**主线程**调用：
- `transform.position = ...`
- `Animator.Play(...)`
- `GameObject.Instantiate(...)`
- `AudioSource.Play(...)`

如果网络线程直接改这些 → Unity 崩溃 / 数据损坏 / 难以复现的 bug。

### 4.2 我们的模式

```
网络线程接收 UDP 包 → 反序列化为事件 → 推入 ConcurrentQueue
                                              ↓
主线程 Update 开头 ← 取出所有 pending 事件 ← 排序后处理
                                              ↓
                                         应用到 Unity 对象
```

每帧固定的"网络 → 主线程"派发点，避免乱序写。

### 4.3 玩家感知

正确实现下，玩家感受不到主线程纪律。错误实现的症状：
- 偶尔的"瞬间瞬移" → 主线程读到半更新数据
- "技能放出来又消失" → GameObject 在错误线程被销毁
- 偶发崩溃 → 触碰 Unity 对象在非主线程

这些 bug 极难复现，**架构期就要严守**。

### 4.4 Worker 线程允许做的事

可以在 worker 线程做（不触 Unity API）：
- 网络收发 / 反序列化
- 数据排序
- DSL VM 解释
- 远端实体插值计算（结果存入 NativeArray，主线程读）
- 文件 I/O / 资源解压

**Burst + Jobs** 系统可以在 worker 线程做大批量计算（如 1000 远端实体的 Hermite 插值），但**不可写 Unity 对象**。

## 5. 资源加载与玩家体验

### 5.1 启动流程

```
玩家启动游戏:
  ├─ Splash screen (~1s)
  ├─ 加载核心引擎 + 必需资源 (~3s)
  ├─ 加载本地配置 + 玩家数据 (~1s)
  ├─ 进入登录界面 (5s 内)
  ↓
玩家登录后:
  ├─ 下载更新（如有，进度条 + 等待时间）
  ├─ 加载主城资源 (~5s, 后台)
  ├─ 进入主城（10s 内总）
```

### 5.2 进入战斗（副本）

玩家点击"进入副本"：
```
  ├─ 显示加载界面（玩家点击瞬间 → 0 ms）
  ├─ 后台：网络请求服务端创建副本 (~100 ms)
  ├─ 后台：下载/加载副本场景 (~3s)
  ├─ 后台：实体预实例化（怪物 prefab）
  ├─ 加载完成（5s 内）
  ├─ 副本启动（玩家进入战斗）
```

**关键**：5 秒内必须进入战斗。超过则玩家流失风险高。

### 5.3 热更

新版本发布：
- 客户端启动后检测版本
- 增量下载（仅变化的 AssetBundle）
- 进度条明确（"下载 2/5 文件，30%"）
- 完成后自动重启

热更过程对玩家**透明**：
- < 5 MB：默默后台下载，玩家不察觉
- 5–50 MB：显示小提示
- > 50 MB：显示进度条 + 提示

### 5.4 资源管理

按场景分包：
- 公共包（角色 / 通用 VFX）
- 主城包
- 各副本包
- 各 PvP 地图包

进入新场景前预加载，平滑过渡。

### 5.5 内存管理

避免 OOM 是底线：
- 主城内存 < 2 GB
- 副本内存 < 4 GB
- 退出场景时释放 AssetBundle

玩家可见症状（如出现）：
- 进入副本闪退（OOM）
- 帧率突然掉到 10 FPS（GC 大暴风）

需要持续监控 + 优化。

## 6. 编辑器工作流

### 6.1 为什么影响玩家体验

工程师/策划在 Unity Editor 内迭代越快 → 玩家感受到的版本质量越高。

### 6.2 关键工作流

| 工作流 | 频率 | 期望耗时 |
|---|---|---|
| 改 C# 代码后运行 | 每天百次 | < 10 s（脚本编译） |
| 改 prefab 后看效果 | 每天数十次 | 即时（Editor 已是所见即所得） |
| 改 Excel 后看效果 | 每天数十次 | < 30 s（DataBuild 增量 + 重启 PIE） |
| 改 native plugin（C++）后看效果 | 每周数次 | < 60 s（C++ 编译 + Unity 加载） |
| 改 shader / material | 每天数次 | < 5 s |
| 加载场景跑战斗测试 | 每天数十次 | < 5 s |

### 6.3 不能容忍的情况

- 改 Excel 后必须重启整个 Unity Editor（极端情况）
- C# 编译 > 30 秒
- Native plugin 改动后需手动复制 .dll
- AssetBundle 重打需要 5 分钟

工具团队持续优化迭代速度——慢的迭代直接降低团队产能。

### 6.4 Editor-Only 功能

许多调试功能仅 Editor 启用：
- Frame Data Editor（参见 `09_tools/FRAME_DATA_EDITOR.md`）
- Combat Replay 查看器
- 网络模拟器（注入延迟/丢包）
- Stat Inspector
- AI 行为树调试器

Player build **不打包**这些，避免被破解利用。

## 7. 玩家可调设置（Settings）

### 7.1 必须支持的设置

| 类别 | 项 |
|---|---|
| **画面** | 分辨率 / 全屏 / 帧率上限 / V-Sync |
| **画质** | 整体档（低/中/高/极高）+ 单项（阴影/贴图/特效/抗锯齿/视野距离） |
| **声音** | 主音量 / 音乐 / 音效 / 语音 / VOIP |
| **控制** | 键位绑定 / 鼠标灵敏度 / 控制器映射 |
| **手感（无障碍）** | Hit pause 强度 / 相机抖动强度 / Slomo 开关 / 闪屏频率 |
| **UI** | UI 缩放 / HUD 元素显隐 / 字体大小 |
| **网络** | 显示 ping / 自动选择服务器 |
| **隐私** | 屏蔽列表 / 显示名称偏好 |

### 7.2 无障碍设置

参见 `COMBAT_FEEL.md §4.7`。

特别强调：
- **Camera shake intensity 0–100%**（晕动症玩家关键）
- **Slomo Enabled** 开关
- **Hit pause strength**
- **闪屏（Flash effect）频率上限**（光敏性癫痫）

这些是行业标准，**不实现会被审核拒绝**。

### 7.3 实时生效

绝大多数设置实时生效，无需重启游戏：
- 切换画质：1 帧内
- 调音量：即时
- 改键位：下次按键即生效

例外：
- 分辨率：可能需要重启 Unity 渲染（< 1 秒）
- 语言：需要重新加载本地化（重启）

## 8. 客户端日志与遥测

### 8.1 玩家不可见但运营可见

客户端运行时上报（取样 / 完整）：
- 帧率分布（按场景）
- 内存占用
- 加载时长
- 关键事件错误（崩溃、严重 bug）
- 网络延迟分布
- 设置使用统计

### 8.2 隐私边界

不上报：
- 玩家 PII（姓名、邮箱、IP 之外的）
- 输入按键序列（除非玩家主动反馈 bug）
- 屏幕截图

**符合 GDPR / 中国个人信息保护法**——必须做对，否则法律风险。

### 8.3 崩溃报告

崩溃时收集：
- 堆栈
- 最近 30 秒事件序列（脱敏）
- 设备信息（型号 / OS / GPU）
- 上传到服务器

工程师据此修复 bug，下版本玩家受益。

## 9. 客户端版本与服务端版本

### 9.1 版本号

```
ClientVersion = "1.4.2.1234"
                 │ │ │  │
                 │ │ │  └── build 号
                 │ │ └────── patch（紧急修复）
                 │ └──────── minor（每月）
                 └────────── major（年度）
```

### 9.2 兼容矩阵

```
ClientVer  | ServerVer | 行为
1.4.0      | 1.4.0     | 完美 ✓
1.4.0      | 1.4.1     | 客户端落后，可玩，提示更新
1.4.0      | 1.5.0     | 客户端必须更新（minor +1 强制）
1.4.0      | 2.0.0     | 客户端必须更新（major +1 强制）
1.5.0      | 1.4.0     | 不可能（客户端总在服务端之后部署）
```

### 9.3 协议版本

每个网络包带 `proto_ver` byte（参见 `OVERVIEW.md §4`）。
- 服务端支持 N 个最近版本（兼容窗口）
- 客户端只用自己版本协议
- 跨版本必须升级客户端

### 9.4 玩家体验

- **常规更新**：客户端启动时检测，下载，5–30 分钟
- **强制更新**：弹框 "新版本可用" + 倒计时（可选，不影响游戏）
- **不兼容**：必须更新才能登录（提供下载链接）

## 10. 调试与作弊检测的客户端侧

### 10.1 客户端反作弊

客户端永远是"敌方阵地"——可以被破解。但仍要做基础防御：
- 完整性校验（DLL hash、关键资源 hash）
- 反调试 / 反内存修改
- VPN / proxy 检测（仅作辅助参考）
- 异常输入频率检测（如每秒 1000 次输入）

**核心原则**：客户端反作弊是**第一道墙**，挡掉脚本小子。**真正防御在服务端**（参见 `08_security/ANTI_CHEAT_STRATEGY.md`，待写）。

### 10.2 玩家不感知正常防御

正常玩家完全感受不到反作弊存在：
- 启动游戏：完整性校验 < 1 秒
- 游戏过程：检测后台运行，开销 < 1% CPU
- 误检率应 < 0.01%（每 10000 玩家 < 1 次误判）

误判会被审核流程修正，玩家不会被永久误封。

### 10.3 调试模式

测试服 / 开发版本启用：
- 详细日志
- 服务器内 console 命令
- 网络延迟模拟
- 调试 UI（HP / buff / 命中点）

正式版完全禁用。

## 11. 跨地区与本地化

### 11.1 多语言

至少支持：
- 简体中文（首发）
- 繁体中文
- 英文
- 日语
- 韩语

每种语言独立资源包。

### 11.2 文化适配

部分内容按地区调整：
- 部分美术（年龄分级要求）
- 部分文案（敏感词）
- 节日活动（中国春节 vs 西方圣诞）

### 11.3 时区

时间显示用本地时区，但游戏内时间（buff 倒计时等）用服务端时间。

每日重置等机制按**服务器时区**（不一定是玩家本地）。

## 12. FAQ 与反模式

### Q1: 用 Unity 是不是手感天花板比 UE5 低？

不是。手感天花板取决于**设计 + 美术 + 工程**的协同投入，引擎只是工具。

Unity 在 ARPG 领域有成功案例（Genshin、HSR）。我们用 Unity 是因为：
- C# 与 Atlas 服务端脚本语言对称（端共享数据类）
- 团队熟悉度（招聘 Unity 工程师容易）
- 生态成熟

UE5 的优势是动画系统更强 + 源码可改，但代价是 C++ 客户端、招聘门槛高。综合权衡选 Unity。

### Q2: 玩家能感觉到 Unity 不做物理判定吗？

不能——只要碰撞体配得对、视觉同步好，玩家根本不知道是不是 PhysX。
反而 PhysX 端不同步会让玩家感觉"挥不到剑"——是糟糕体验。

### Q3: 自研 Native plugin 万一崩溃整个游戏挂了怎么办？

崩溃必然存在（任何代码都可能 bug）。应对：
- Plugin 内部异常 → 捕获 + 上报 + 玩家友好提示（"发生错误，请重试"）
- 完整崩溃 → Unity 自动重启
- 严重 bug 走 hotfix 流程，不让玩家长期受影响

不要因为"怕崩溃"而避免 native plugin——确定性收益远大于稳定性风险。

### Q4: 为什么不直接用 Unity Netcode for GameObjects？

它是为休闲多人设计的，不适合 MMOARPG：
- 假设客户端权威（基础前提）
- 性能不足以支撑 5000 并发
- 缺乏严格端同保证

我们的协议层是为 Atlas 量身设计——参见 `OVERVIEW.md §6.2`、`MOVEMENT_SYNC.md`。

### Q5: AssetBundle 比 Addressables 哪个好？

Addressables 是 Unity 推荐的更现代方案，是 AssetBundle 的高级包装。
我们用 Addressables（功能更完善），但本文用 "AssetBundle" 作为通用名词指代。

### Q6: Unity 升级版本会不会破坏端同？

会有风险——`Math.Sin` 等可能在 Unity 升级时改实现。所以：
- 关键路径用自研 `Atlas.CombatCore.Math`（端共享 C++）
- 不依赖 Unity 内置数学函数做权威计算
- 升级前在测试服跑完整端同回归

每年大版本升级（如 Unity 6 → 7）需要充分测试。

### Q7: 客户端能不能完全"信任"服务端的事件顺序？

99.9% 时间可以——服务端发的事件按 server tick 排序。极少数 0.1% 网络异常时事件迟到，客户端按 tick 重排（见 `COMBAT_EVENT_ORDERING.md §6`）。

玩家可能感知"这个 buff 应该在我那个技能命中后才有"等，但极少（< 0.1% 战斗）。

### Q8: 单玩家最低硬件要求是什么？

P3 阶段定基线：
- CPU: i5 8400 / Ryzen 5 2600（2018 中端）
- GPU: GTX 1060 / RX 580（同期中端）
- RAM: 8 GB
- 硬盘: SSD 推荐

低于此玩家可能体验下降但不卡死。最低绝对要求 4 GB RAM + 集成显卡（极低画质 30 FPS）。

### Q9: 客户端能感知服务端在做 cell 迁移吗？

不能。BaseApp 是稳定锚点：
- 客户端只连 BaseApp（不直接连 CellApp）
- BaseApp 内部把请求转发到正确 CellApp
- Cell 迁移期间 BaseApp 转发可能短暂延迟（< 50ms），玩家感觉一次轻微 ping 抖动

参见 `CELL_ARCHITECTURE.md §4.3`。

### Q10: PvP arena 为什么 30 Hz 不上 60 Hz？

P3 阶段 30 Hz 已通过盲测达标。60 Hz 边际收益：
- 视觉差异：玩家几乎不察觉（动画插值能补）
- 输入延迟：从 33ms 降到 17ms，但本地预测已 0ms
- 服务端成本：CPU 翻倍

如 P3 验证 30 Hz 不够再升 60 Hz。先用 30 Hz 节省成本。

- ❌ Editor 工作流忽略（开发慢 = 版本质量低）
